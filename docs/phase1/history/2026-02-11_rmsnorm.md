# 2026-02-11: RMSNorm CUDA 커널 구현

## 작업 개요

Phase 1의 세 번째 커널인 **RMSNorm**과 **Fused Add+RMSNorm**을 구현하였다.
RoPE 커널과 동일한 아키텍처 패턴(순수 CUDA 헤더 + PyTorch wrapper + Python 래퍼 + 테스트)을 따르며,
CUB `BlockReduce`를 사용한 효율적인 병렬 리덕션을 도입하였다.

---

## 배경

### RMSNorm이란

RMSNorm (Root Mean Square Layer Normalization)은 LLaMA, Mistral 등 최신 LLM에서
표준 LayerNorm 대신 사용하는 정규화 방식이다.

```
RMSNorm(x) = x / sqrt(mean(x²) + ε) × γ
```

LayerNorm과 달리 **mean centering (x - μ)**을 하지 않아 리덕션이 1회만 필요하며,
실험적으로 동등한 학습 품질을 달성한다.

### Fused Add+RMSNorm이 필요한 이유

Transformer 블록에서 RMSNorm은 항상 잔차 연결(residual connection)과 함께 사용된다:

```
residual = hidden_states + attention_output   ← 덧셈 (커널 1)
hidden_states = RMSNorm(residual)             ← 정규화 (커널 2)
```

별도 커널 2개를 사용하면 `residual`을 글로벌 메모리에 썼다가 바로 다시 읽어야 한다.
Fused 커널은 이 두 연산을 하나로 합쳐 **글로벌 메모리 읽기 1회를 절약**한다.

## 설계 결정

### Approach B (교육적이되 성능도 고려) 채택

| 선택지 | 설명 | 채택 여부 |
|--------|------|-----------|
| A: 단순 | 수동 warp reduction, 벡터화 없음 | ✗ |
| **B: 균형** | CUB BlockReduce, 2D 텐서 전용, 스칼라 루프 | **✓** |
| C: vLLM 풀 | 벡터화, _f16Vec, 다차원 텐서, SFINAE 분기 | ✗ |

**이유**: CUB는 CUDA Toolkit에 포함된 표준 라이브러리로 추가 의존성이 없으며,
블록 리덕션의 실제 표준 구현 방식을 학습할 수 있다.
2D 텐서만 지원하면 코드가 간결하면서도 LLaMA 유스케이스를 완전히 커버한다.

### CUB BlockReduce 사용 방식

```cpp
using BlockReduce = cub::BlockReduce<float, 1024>;
__shared__ typename BlockReduce::TempStorage reduceStore;
variance = BlockReduce(reduceStore).Reduce(variance, CubAddOp{}, blockDim.x);
```

- 템플릿 파라미터 `1024`는 최대 지원 스레드 수 (컴파일타임 상수 필요)
- 실제 블록 크기가 더 작을 수 있으므로 `blockDim.x`를 유효 스레드 수로 전달
- vLLM과 동일한 패턴

### CUB 버전 호환

CUB 2.0.8 이상에서 `cub::Sum()`이 `cuda::std::plus<>`로 변경되었다.
`cub_compat.h` 헤더를 신규 생성하여 버전에 따라 `CubAddOp` 타입을 자동 선택하도록 했다.

## 구현 상세

### 커널 알고리즘 (Two-Pass)

```
Grid: (num_tokens,)  — 각 블록이 하나의 토큰(행) 처리
Block: (block_size,) — 스레드들이 hidden_size 원소를 협력 처리

Pass 1: 제곱합 계산
  ┌─ 각 스레드가 stride 패턴으로 hidden_size 순회
  ├─ float로 변환하여 variance += x * x 누적 (수치 안정성)
  ├─ CUB BlockReduce로 블록 전체 합산
  └─ Thread 0: s_variance = rsqrtf(variance / hidden_size + ε)
     → __shared__에 저장 → __syncthreads()로 브로드캐스트

Pass 2: 정규화 적용
  └─ output[i] = (float(input[i]) * s_variance) * weight[i]
```

### Fused 커널 차이점

Pass 1에서 덧셈과 제곱합 누적을 동시에 수행:

```cpp
float sum = inp + res;       // 레지스터에서 덧셈
variance += sum * sum;        // 바로 제곱합 누적
residual[idx] = (scalar_t)sum; // residual 업데이트
```

이후 Pass 2에서 residual에서 읽어 input에 정규화 결과를 저장한다.

### 블록 크기 휴리스틱

```
max_block_size = (num_tokens < 256) ? 1024 : 256
block_size = min(hidden_size, max_block_size)
```

- 토큰이 적으면 → 큰 블록으로 SM 활용도 극대화
- 토큰이 많으면 → 작은 블록으로 SM 동시성 증대 (레이턴시 은닉)
- vLLM과 동일한 휴리스틱

## 허용 오차 참고

RMSNorm은 CUB BlockReduce를 통한 **병렬 리덕션**을 사용하므로,
PyTorch의 순차적 리덕션과 부동소수점 누적 순서가 다르다.
따라서 RoPE처럼 bit-exact 비교(atol=0, rtol=0)는 불가능하며,
dtype별 허용 오차를 사용한다:

| dtype | atol | rtol |
|-------|------|------|
| float32 | 1e-5 | 1e-5 |
| float16 | 1e-3 | 1e-3 |
| bfloat16 | 1e-2 | 1e-2 |

단, `test_fused_residual_update`에서 residual 업데이트는 리덕션과 무관한
단순 덧셈이므로 **bit-exact (atol=0, rtol=0)** 검증이 가능하다.

## 교육 문서 내용 요약

### layernorm_kernels.cuh (커널 헤더)

1. **RMSNorm vs LayerNorm 비교**: 리덕션 횟수 차이, mean centering 불필요성
2. **fp32 누적의 필요성**: bf16의 유효 자릿수 한계와 catastrophic cancellation
3. **CUB BlockReduce 동작 원리**: warp shuffle → warp 간 공유 메모리 리덕션
4. **`__shared__` 브로드캐스트 패턴**: Thread 0 계산 → 동기화 → 전체 읽기
5. **Fused 커널의 메모리 절약 원리**: 글로벌 메모리 읽기 1회 절약의 정량적 분석

### layernorm_kernels.cu (PyTorch wrapper)

1. **LLM에서 정규화가 필요한 이유**: Internal Covariate Shift, Gradient 문제
2. **LLaMA의 Pre-Norm 구조**: Post-Norm과의 차이
3. **RMSNorm 수학**: rsqrt 사용 이유, GPU SFU 하드웨어 명령어
4. **LayerNorm과의 상세 비교**: 수식, 리덕션 횟수, 파라미터 수
5. **Fused 커널의 메모리 대역폭 분석**: 별도 커널 vs Fused 커널의 메모리 접근 비교
6. **블록 크기 선택 휴리스틱**: 토큰 수에 따른 SM 활용도/동시성 트레이드오프

## 핵심 학습 포인트

1. **CUB는 CUDA 생태계의 표준 리덕션 라이브러리**이다. 수동 warp shuffle 코딩 대신 CUB를 사용하면 정확하고 효율적인 리덕션을 쉽게 구현할 수 있다.

2. **Half precision에서는 반드시 fp32로 누적**해야 한다. bf16의 유효 자릿수가 약 2.4자리밖에 되지 않아, hidden_size=4096개의 값을 bf16으로 누적하면 작은 값이 소실된다.

3. **커널 퓨전의 본질은 메모리 대역폭 절약**이다. GPU는 연산보다 메모리 접근이 병목인 경우가 많으므로, 불필요한 글로벌 메모리 왕복을 제거하는 것이 가장 효과적인 최적화이다.

4. **병렬 리덕션은 순차 리덕션과 결과가 다를 수 있다**. 부동소수점 덧셈은 결합법칙이 성립하지 않으므로, 누적 순서에 따라 미세한 차이가 발생한다. 테스트에서 적절한 허용 오차를 설정해야 한다.
